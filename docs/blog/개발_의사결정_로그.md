# Polling In Run 개발 의사결정 로그

> 목적: 개발 중 고민한 선택지, 판단 이유, 결과를 모아 블로그와 포트폴리오 회고 재료로 사용한다.

## 글로 풀어낼 핵심 주제

- 러닝 앱에서 기록보다 편의시설 정보가 중요할 수 있다는 문제 발견
- 공공데이터를 지도 UI에 맞게 정규화하는 과정
- 프론트엔드가 외부 데이터를 직접 읽는 방식과 FastAPI가 공통 모델로 변환하는 방식의 차이
- Kakao Maps JavaScript SDK의 웹 도메인 제약과 Capacitor iOS WebView 한계
- 풀스택 확장보다 local-first iPhone prototype으로 범위를 줄인 이유
- 로그인/회원가입을 줄이고 마이 페이지를 로컬 대시보드 후보로 바꾼 이유
- 지도 전략을 Apple MapKit native 우선으로 재선정한 이유

## 결정 1. 공공데이터는 프론트에서 직접 읽지 않고 FastAPI에서 정규화한다

### 고민

음수대와 화장실 데이터는 서로 다른 출처, 형식, 컬럼 이름, 좌표 품질을 가진다. 프론트에서 각각의 데이터를 직접 읽으면 지도 화면 코드가 데이터 소스의 사소한 차이에 계속 흔들린다.

### 선택지

- 프론트에서 각 공공데이터 API/XLSX를 직접 읽는다.
- FastAPI가 여러 원본을 수집하고 공통 `Facility` 모델로 변환한다.

### 판단

웹 MVP에서는 FastAPI 정규화를 선택했다. 프론트는 지도와 상호작용에 집중하고, FastAPI는 외부 데이터 차이를 흡수한다.

### 결과

- 음수대와 화장실을 같은 마커 모델로 표시할 수 있었다.
- 현재 위치와 지도 영역 기반 시설 조회가 쉬워졌다.
- 단, iPhone local-first 방향으로 바꾸면서 장기적으로는 정규화된 JSON을 앱 번들에 포함하는 구조가 필요해졌다.

관련 문서: [`주변 편의시설`](../features/facility-search.md)

## 결정 2. Kakao Maps는 웹 MVP에는 좋지만 iPhone 앱 전략으로는 한계가 있다

### 고민

Kakao Maps JavaScript SDK는 웹에서 빠르게 지도와 마커를 구현하기 좋았다. 하지만 iPhone에 Capacitor 앱으로 설치하자 `capacitor://localhost` origin에서 SDK 도메인 검증 문제가 발생했다.

### 확인한 문제

- 모바일 브라우저나 Vite dev server URL에서는 Kakao Developers Web 플랫폼 도메인을 맞추면 동작한다.
- Capacitor 정적 앱은 `capacitor://localhost`에서 실행되어 일반 웹 도메인 등록 전략과 맞지 않는다.
- Vite dev server URL을 `server.url`로 로드하면 우회할 수 있지만, 이는 개발 디버깅용이지 실사용 앱 전략이 아니다.

### 판단

웹 MVP는 Kakao Maps를 유지하되, iPhone local-first 앱의 지도 전략은 다시 고른다.

관련 문서: [`카카오맵과 현재 위치`](../features/kakao-map-and-location.md), [`지도 전략`](../features/map-strategy.md)

## 결정 3. iPhone 앱 지도는 Apple MapKit native를 1차 후보로 둔다

### 선택지

- Apple MapKit native
- Apple MapKit JS
- Google Maps SDK for iOS
- Leaflet + OpenStreetMap
- 외부 지도 앱 연동

### 판단

현재 목표가 내 iPhone에서 안정적으로 쓰는 local-first prototype이라면 Apple MapKit native가 가장 자연스럽다. 위치 권한, 마커, iOS 앱 경험과 잘 맞는다.

다만 React WebView 안에 바로 넣기는 어렵고 Capacitor native bridge 또는 Swift 구현이 필요하다. 그래서 다음 단계는 구현 경로를 조사하는 것이다.

### 보류한 것

- Google Maps SDK for iOS는 기능은 좋지만 billing과 API key 관리가 필요해 리소스 최소화 방향과 덜 맞다.
- Leaflet + OpenStreetMap은 WebView 실험에는 좋지만 공개 타일 정책과 iOS 앱다운 경험을 따로 고려해야 한다.
- 외부 지도 앱 연동은 단기적으로 계속 유지한다.

관련 문서: [`지도 전략`](../features/map-strategy.md)

## 결정 4. 풀스택을 계속 키우지 않고 local-first로 범위를 줄인다

### 고민

초기에는 FastAPI, Supabase Auth, PostgreSQL, 계정 삭제까지 풀스택 구조를 실험했다. 하지만 이 프로젝트의 핵심 목적은 대외 서비스 운영이 아니라, 러닝 중 필요한 편의 정보를 내 iPhone에서 빠르게 확인하는 경험을 검증하는 것이다.

### 판단

현재 MVP는 local-first iPhone prototype으로 재정의한다.

```text
iPhone App
├─ 지도와 현재 위치
├─ 로컬 시설 데이터
├─ 로컬 러닝 기록
├─ 마이 페이지 사용자 정보·설정
└─ 추후 확장
   ├─ FastAPI 공공데이터 갱신
   └─ Supabase Auth/PostgreSQL 동기화
```

### 결과

- 서버 배포와 계정 운영 부담을 줄였다.
- 로그인 없이 러닝 기록을 로컬에 저장하고 조회하는 방향으로 바꿨다.
- Supabase Auth는 버린 것이 아니라 멀티 디바이스 동기화가 필요할 때의 확장안으로 남겼다.

관련 문서: [`Local-first iPhone Prototype`](../features/local-first-iphone-prototype.md), [`인증과 설정`](../features/auth-and-settings.md)

## 결정 5. 마이 페이지는 로그인 화면이 아니라 러닝 대시보드 후보로 둔다

### 고민

처음에는 마이 페이지에 ID/PW 로그인, 회원가입, 회원 탈퇴를 넣었다. 하지만 local-first MVP에서는 계정 기능보다 내 기록을 이해하는 화면이 더 가치 있다.

### 현재 방향

마이 페이지는 로컬 프로필과 앱 상태를 보여주는 화면으로 축소했다. 이후 사용자와 상의해 러닝 대시보드로 확장한다.

### 대시보드 후보

- 총 뛴 거리
- 총 러닝 횟수
- 최근 러닝 날짜와 거리
- 월간 거리
- 최장 러닝 거리
- 평균 페이스 추이
- 목표 설정
- 이번 주/이번 달 목표 대비 진행률
- 목표 대비 비교 그래프
- 러닝 기록 리셋
- 달력 기반 러닝 날짜 표시
- 거리 또는 횟수 그래프

### 기록 페이지로 분리한 기능

- 메모가 있는 러닝만 모아보기
- 거리, 시간, 페이스 기준 정렬
- 월별 기록 필터
- 기록 상세의 경로 지도 시각화

관련 문서: [`마이 페이지`](../pages/my-page.md), [`기록`](../pages/records.md)
