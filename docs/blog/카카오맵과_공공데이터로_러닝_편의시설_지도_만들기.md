# 카카오맵과 공공데이터로 러닝 편의시설 지도 만들기

> 상태: 초안
> 프로젝트: Polling In Run
> 주제: Kakao Maps, FastAPI, 공공데이터, 지도 SDK 도메인, 길찾기 의사결정

## 글의 목적

Polling In Run은 러닝 중 가까운 음수대와 화장실을 빠르게 찾기 위한 모바일 중심 웹앱이다.

이 글은 단순히 지도 SDK를 붙인 기록이 아니라, 프론트엔드 개발자가 공공데이터와 백엔드 API를 함께 다루면서 어떤 판단을 했는지 정리하기 위한 초안이다.

핵심 질문은 네 가지였다.

1. 지도는 어떤 API를 사용할 것인가?
2. 공공데이터는 프론트에서 직접 읽을 것인가, 백엔드에서 정규화할 것인가?
3. 로컬 개발과 모바일 확인에서 카카오 지도 도메인 문제를 어떻게 다룰 것인가?
4. 길찾기는 내 서비스 안에서 구현할 것인가, 카카오맵으로 넘길 것인가?

## 문제 정의

러닝 앱이라고 하면 보통 기록 측정, 페이스, 칼로리, 랭킹을 먼저 떠올린다. 하지만 실제로 뛰다 보면 더 즉각적인 문제가 있다.

- 물이 필요하다.
- 화장실이 필요하다.
- 러닝 중에는 긴 검색 과정을 거치기 어렵다.
- 모바일에서 몇 초 안에 주변 시설을 확인해야 한다.

그래서 Polling In Run의 첫 번째 MVP는 "러닝 기록"보다 "러닝 중 편의시설 확인"에 집중했다.

## 선택한 기술 구조

현재 구조는 다음과 같다.

```text
React + TypeScript
→ FastAPI
→ 서울 열린데이터광장 음수대 API
→ 공통 Facility 모델
→ Kakao Map 마커
```

프론트엔드는 지도와 사용자 상호작용에 집중하고, FastAPI는 외부 데이터를 수집하고 공통 모델로 변환한다.

처음에는 프론트에서 공공데이터를 직접 호출하는 방식도 생각할 수 있었다. 하지만 실제로는 FastAPI를 중간에 두는 편이 더 낫다고 판단했다.

## 왜 FastAPI에서 데이터를 정규화했나

공공데이터는 원본마다 컬럼 이름, 좌표 형식, 갱신주기, 인증 방식이 다르다.

예를 들어 음수대 데이터는 서울 열린데이터광장 Open API로 받을 수 있고, 화장실 데이터는 현재 확인한 출처 기준으로 승인형 ZIP/XLSX 다운로드 데이터에 가깝다.

프론트가 각각의 원본 형식을 직접 알게 만들면 화면 코드가 데이터 출처에 강하게 묶인다.

그래서 FastAPI에서 다음과 같은 공통 모델로 변환했다.

```ts
type Facility = {
  id: string
  type: 'water' | 'restroom'
  name: string
  latitude: number
  longitude: number
  address: string
  road_address: string | null
  opening_hours: string | null
  source: string
  distance_m: number | null
}
```

이렇게 하면 React는 원본이 서울 Open API인지, XLSX 파일인지 몰라도 된다. 지도는 항상 `Facility` 배열만 받아 마커를 그리면 된다.

## 음수대 데이터 연결

음수대는 서울 열린데이터광장의 `TbViewGisArisu` 데이터를 사용했다.

FastAPI는 API 키를 서버 환경변수로 관리하고, 앱 시작 후 첫 조회 시 데이터를 가져와 메모리에 캐시한다. 이후 현재 위치와 반경이 전달되면 거리 계산과 정렬을 수행해 프론트에 반환한다.

중요했던 점은 API 키를 프론트 `.env.local`에 두지 않는 것이었다. 브라우저에 노출되어도 되는 Kakao JavaScript 키와 달리, 서버에서만 써야 하는 키는 `apps/api/.env`에서 관리했다.

## 카카오 지도 SDK 도메인 문제

로컬에서 카카오 지도가 나오지 않는 문제가 있었다.

원인은 코드 오류가 아니라 Kakao Developers에 등록한 Web 플랫폼 도메인과 실제 접속 주소가 달랐기 때문이다.

예를 들어 아래 주소들은 개발자에게는 비슷해 보이지만, 카카오 입장에서는 서로 다른 도메인이다.

```text
http://localhost:5173
http://127.0.0.1:5173
http://192.168.45.3:5173
```

PC에서 `localhost`를 등록해두고 `127.0.0.1`로 접속하면 지도가 실패할 수 있다. 모바일에서 Mac의 개발 서버를 보려고 `192.168.x.x` 주소로 접속해도 마찬가지다.

그래서 개발 환경에서는 `127.0.0.1` 접속을 `localhost`로 리다이렉트하는 보조 장치를 추가했다.

단, 이 리다이렉트는 운영 환경에서 쓰는 해결책이 아니다. 정식 배포에서는 실제 배포 도메인을 Kakao Developers에 등록해야 한다.

```text
로컬 PC 개발: http://localhost:5173
모바일 실기기 테스트: http://192.168.x.x:5173
정식 배포: https://서비스도메인
```

## 마커 클릭과 상세 카드

지도에 음수대 마커가 보이기 시작하자 다음 문제는 "사용자가 이 시설이 어디인지 어떻게 확인할 것인가"였다.

그래서 마커를 클릭하면 지도 위에 상세 카드를 띄우도록 했다.

카드에는 다음 정보를 보여준다.

- 시설 종류
- 현재 위치 기준 거리
- 시설명
- 주소
- 운영시간
- 길찾기 버튼

이 방식은 모바일에서 화면 전환을 늘리지 않고, 지도 맥락 안에서 빠르게 정보를 확인할 수 있다는 장점이 있다.

## 길찾기는 왜 카카오맵으로 넘겼나

처음에는 내 서비스 안에서 길찾기를 바로 실행하고 싶었다. 하지만 여기에는 두 단계가 있다.

첫 번째는 카카오맵 길찾기 URL로 보내는 방식이다.

```text
내 서비스
→ Kakao Map 길찾기 URL
→ 카카오맵 웹 또는 앱
```

두 번째는 내 서비스 안에서 경로 계산까지 하는 방식이다.

```text
현재 위치
→ 길찾기 API 호출
→ 경로 좌표 수신
→ 지도 위에 Polyline 표시
```

두 번째 방식은 UX가 좋아 보이지만, 별도 길찾기 API와 REST API 키가 필요하고 비용과 쿼터를 고려해야 한다. 또한 카카오모빌리티의 대표 길찾기 API는 자동차 길찾기 중심이라 러닝이나 도보 경험과 맞지 않을 수 있다.

그래서 MVP에서는 카카오맵 URL로 넘기는 방식을 선택했다.

이 결정의 장점은 명확하다.

- 구현이 빠르다.
- 비용 부담이 작다.
- 사용자는 익숙한 카카오맵에서 길찾기를 이어갈 수 있다.
- 내 서비스는 "주변 편의시설 발견"이라는 핵심 가치에 집중할 수 있다.

나중에 고도화한다면 "앱 안에서 경로 보기"를 별도 실험 기능으로 추가할 수 있다.

- 현재 위치 반경 필터와 거리순 정렬
- 현재 지도 영역 기반 시설 조회
- 마커 클러스터링
- 음수대/화장실 필터 칩
- 음수대/화장실 SVG 마커
- 마커 클릭 상세 카드
- 카카오맵 길찾기 URL 연결
- 러닝 시작·일시정지·종료 상태 모델

## 회고

이번 작업에서 가장 크게 배운 점은 지도 기능은 단순히 SDK를 붙이는 일이 아니라는 것이다.

실제로는 다음 요소들이 함께 맞물린다.

- 브라우저 위치 권한
- 지도 SDK 도메인 제한
- 공공데이터의 품질과 좌표 형식
- 서버 환경변수와 API 키 보안
- 프론트엔드 화면 상태
- 모바일에서의 조작 흐름
- 비용과 쿼터를 고려한 기능 범위 조절

Polling In Run은 아직 작은 토이프로젝트지만, 이 과정은 풀스택 개발의 좋은 축소판이었다.

프론트엔드 개발자로서 화면을 만드는 것에 그치지 않고, 데이터가 어디서 오고, 어떤 형태로 정리되고, 어떤 운영 제약을 갖는지까지 함께 설계해야 했다.

이 프로젝트의 다음 목표는 화장실 실제 데이터를 연결해 "샘플 지도"에서 "실제 러닝 편의 지도"로 한 단계 더 나아가는 것이다.
