# 기록 저장

## 해결하려는 문제

완료한 러닝 기록과 메모를 다시 확인할 수 있도록 저장해야 한다.

## 선택 방향

초기 데이터 모델 실험은 브라우저 `localStorage`로 빠르게 수행하고, 사용자별 기록과 배포가 필요해질 때 Supabase PostgreSQL을 연결한다.

## SQLite와 Supabase PostgreSQL 비교

### SQLite

- 장점: 설치가 간단하고 로컬 API·데이터 모델 실험이 빠르다.
- 단점: 여러 사용자와 배포 환경에서 확장하기 어렵다.

### Supabase PostgreSQL

- 장점: 실제 배포, 사용자별 데이터, 관리 화면과 백업에 적합하다.
- 단점: 초기 설정과 인증·권한 설계가 추가된다.

## 추천

데이터 모델을 빠르게 학습해야 한다면 SQLite로 짧게 실험할 수 있다. 다만 최종 포트폴리오 구조는 이미 선택한 Supabase PostgreSQL을 사용한다.

## 기록 모델 초안

- `id`
- `savedAt`
- `distanceM`
- `elapsedMs`
- `pace`
- `routePointCount`
- `memo`

향후 Supabase 연결 시 추가할 필드:

- 사용자 ID
- 시작·종료 시각
- 경로 좌표
- 사진 URL

## 구현 기록

### 2026-06-21

- 러닝 결과 화면에서 메모를 입력할 수 있다.
- `localStorage`의 `polling-in-run.records.v1` 키에 러닝 기록을 저장한다.
- 저장 데이터에는 거리, 시간, 평균 페이스, GPS 포인트 수, 메모, 저장 시각이 포함된다.
- 저장 실패 시 메모 입력값은 화면에 유지하고 오류 안내를 표시한다.
- 기록 목록과 상세 초안을 연결했다.
- Supabase 저장은 다음 단계로 분리했다.

## 완료 조건

- 로컬 MVP에서는 완료한 러닝 결과와 메모를 저장할 수 있다.
- 로컬 MVP에서는 저장한 기록을 목록과 상세로 다시 확인할 수 있다.
- 저장 실패 시 사용자가 입력한 메모를 잃지 않는다.
- Supabase 전환 시 필요한 기록 모델 필드를 문서화한다.
- 사용자 인증 추가 후 다른 사용자의 기록에 접근할 수 없다.
