# 주변 편의시설

## 해결하려는 문제

러닝 전후와 러닝 중에 가까운 공공 음수대와 화장실을 빠르게 찾아야 한다.

## 구현 방향

FastAPI가 여러 공공데이터 원본을 공통 시설 모델로 정규화하고, React는 현재 위치 주변의 정규화된 시설만 표시한다.

```text
공공데이터 원본
→ FastAPI 정규화·거리 필터
→ 공통 시설 API
→ React 지도 마커
```

## 확인한 데이터 소스

### 서울시 공원 음수대

- 서울 열린데이터광장 Open API `TbViewGisArisu`
- 매주 갱신되며 위도·경도가 제공된다.
- Polling In Run의 음수대 핵심 데이터 소스로 사용한다.
- API가 HTTP로 제공되므로 FastAPI가 호출하고 결과를 캐시한다.

상세: [`서울시 공원 음수대`](../data-sources/seoul-park-water-fountains.md)

### 서울시 공중화장실

- 공간데이터마켓의 승인형 ZIP/XLSX 다운로드 데이터다.
- 공개 샘플에는 화장실명, 주소, 개방시간, 위도·경도가 포함된다.
- 실제 DBF 원본에는 4,799건의 행 데이터가 있고, 이 중 4,793건을 공통 시설 모델로 정규화한다.
- 배포주기가 1년이므로 초기 시드와 모델 검증에 사용하고, 운영용 최신 데이터 출처를 추가 조사한다.

상세: [`서울시 공중화장실`](../data-sources/seoul-public-restrooms.md)

## FastAPI를 두는 이유

### 장점

- 서로 다른 공공데이터 형식을 프론트엔드에서 숨길 수 있다.
- 외부 API 키와 데이터 정제 규칙을 서버에서 관리할 수 있다.
- 데이터 출처가 변경되어도 프론트엔드 계약을 유지하기 쉽다.

### 단점

- 서버 배포와 장애 지점이 추가된다.
- 초기에는 정적 데이터만 표시하는 방식보다 구현량이 많다.

## 공통 시설 모델 초안

- `id`
- `source`
- `source_id`
- `type`: `water` 또는 `restroom`
- `name`
- `latitude`
- `longitude`
- `address`
- `road_address`
- `opening_hours`
- `details`
- `source_updated_at`
- `synced_at`

## 주변 시설 API 초안

```http
GET /api/facilities?latitude=37.5665&longitude=126.9780&radius_m=3000&type=water,restroom
```

응답은 현재 위치와의 거리순으로 정렬한다. 프론트엔드는 서울 Open API나 다운로드 파일의
원본 컬럼을 알 필요 없이 공통 시설 모델만 사용한다.

## 추천 구현 순서

1. 샘플 음수대·화장실 데이터를 공통 모델로 변환한다. 완료
2. FastAPI 정적 시설 API로 지도 마커 UI를 완성한다. 완료
3. 현재 위치 반경 필터와 거리순 정렬을 구현한다. 완료
4. 서울 음수대 Open API 수집기와 캐시를 연결한다. 완료
5. 승인받은 화장실 파일 인제스트 스크립트를 만든다. 완료
6. 데이터 품질 검증을 추가한다.

## 샘플 수직 기능 기록

- FastAPI의 `/api/facilities`가 음수대와 화장실을 같은 모델로 반환한다.
- React는 원본 필드명을 알지 않고 공통 모델만 조회한다.
- 첫 렌더링에서 React Query가 `/api/facilities`를 호출한다.
- 좌표 없이 호출하면 공통 모델의 샘플 배열을 반환한다.
- 좌표와 반경을 전달하면 FastAPI가 거리 계산·반경 필터·거리순 정렬을 수행한다.
- React는 위치 확인이 끝난 뒤 좌표를 전달해 첫 시설 요청부터 가공된 주변 데이터를 받는다.
- 위치 권한을 사용할 수 없으면 좌표 없이 전체 샘플 시설을 요청한다.
- 현재 지도 영역 기반 필터는 아직 구현되지 않았다.
- 카카오맵과 필터 칩에는 음수대 물방울, 화장실 픽토그램 원형 마커를 표시한다.
- 필터 버튼으로 시설 종류별 마커를 켜고 끌 수 있다.
- 음수대는 서울 열린데이터광장 실제 데이터를 사용한다.
- 화장실은 공간데이터마켓 DBF 원본의 실제 데이터를 사용한다.
- 실제 시설이 밀집된 지역에서는 마커가 겹치므로 데이터 연결 후 클러스터링을 추가한다.

## 마커 표시 성능 전략

소량의 샘플 마커는 모두 렌더링해도 문제가 없다. 하지만 서울 음수대 약 1,685건과
화장실 데이터를 모두 DOM 기반 커스텀 마커로 동시에 표시하면 렌더링 비용, 겹침,
시각적 복잡도가 커진다.

추천 단계:

1. 현재 샘플 단계에서는 모든 결과 마커를 표시한다.
2. `시설 전체 보기` 버튼은 현재 조회 결과의 범위에 지도를 맞춘다.
3. 실제 데이터 단계에서는 현재 지도 화면 영역 안의 시설만 FastAPI에 요청한다.
4. 축척이 넓을 때는 가까운 마커를 클러스터로 묶는다.
5. 확대하면 클러스터를 개별 시설 마커로 풀어 보여준다.

모든 마커를 무조건 숨기는 것이 아니라, 사용자가 보고 있는 범위와 축척에 맞게 필요한
마커를 보여주는 방식이다.

## 첫 렌더링 API 응답 원칙

첫 화면에 필요한 데이터는 FastAPI가 화면에서 바로 사용할 수 있는 형태로 가공해 반환한다.
프론트엔드는 외부 원본 필드 변환, 거리 계산, 데이터 품질 판정을 담당하지 않는다.

```text
외부 원본 수집
→ FastAPI 공통 모델 정규화
→ 현재 위치 또는 지도 영역 필터
→ 거리 계산과 정렬
→ 필요한 필드만 응답
→ React Query 캐시와 지도 렌더링
```

다만 아이콘 SVG나 버튼 표시 상태처럼 화면 표현에만 필요한 값은 프론트엔드가 결정한다.

## 완료 조건

- 현재 위치 반경 안의 시설을 조회할 수 있다.
- 물과 화장실 마커를 구분할 수 있다.
- 데이터가 없거나 조회에 실패한 상태를 안내한다.
- 원본 출처와 마지막 동기화 시점을 추적할 수 있다.
